\begin{abstract}
When a long-lived assistant answers from memory, the load-bearing question is often not ``was retrieval relevant?'' but ``can anyone prove, later, why that memory surfaced?'' We call this task \emph{recall forensics}: from a durable database and a published evidence format, an auditor reconstructs a logged result's ranking inputs, provenance commitment, lifecycle admissibility, and integrity evidence. We describe the governed-memory architecture in \system{} v0.3.0 and evaluate additional forensics bindings in a content-addressed post-release source snapshot. Across 14 turns containing a correction, quarantine, explicit promotion, deletion, and later graph rebuild, a standard-library auditor recomputed 14/14 ranking decisions and reconstructed 68/68 returned-result pairs. Exact engine re-execution is reported only for 11/11 turns whose canonical inputs remained unchanged; three earlier turns are excluded because retaining deleted plaintext for replay would contradict the deletion contract. The evaluator also detected 75/75 selected single-row mutations. Tail truncation remained undetectable from the chain alone (0/5) and detectable with an anchored checkpoint (5/5).

The architecture behind this capability treats memory as governed local state. Canonical episodes and records carry provenance, categorical trust, lifecycle state, and append-only transition evidence on one per-subject SHA-256 chain that also records graph, retrieval, approval, and action transitions. Retrieval uses bounded SQLite FTS5 candidates and an optional, derived entity-relation graph, fused deterministically; the graph adds explainable multi-hop joins while remaining disposable and rebuildable from canonical records. No embedding model or vector database is used --- a deliberate trade that avoids preserving an additional learned model, tokenizer, and vector-index artifact for replay, at the cost of some paraphrase recall. Exact-plan HMAC approvals, root-confined filesystem adapters, precondition revalidation, postcondition checks, compensation, and explicit uncertain states keep recalled information distinct from authority to act.

We state platform boundaries explicitly: on macOS the desktop launcher seals the database with OpenSSL AES-256-CBC and HMAC-SHA256 under a Keychain-held secret while the live database remains plaintext during operation; Linux and Windows rely on operating-system disk encryption. A reproducible 10,002-record laptop probe additionally demonstrates candidate-bounded recall and a two-hop graph result; it is a scale sanity check, not a quality or million-record validation.
\end{abstract}
