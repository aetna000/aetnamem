The base commit alone does **not** contain the forensics changes. The file
manifest identifies the evaluated working-tree bytes and is checked by
`validate_evidence.py`. This avoids attributing development evidence to the
clean v0.3.0 release.

The scale-probe evidence is `governed-memory-probe.json`. It is generated by:

```bash
python3 paper/scripts/run_governed_memory_probe.py \
  --records 10000 \
  --iterations 25 \
  --candidate-cap 200 \
  --output paper/data/governed-memory-probe.json
```

The JSON is a single-host timing observation. Its rank, cap, and path fields
support the narrow mechanism described in the paper; the latency fields are
not portable performance guarantees. The probe uses in-memory SQLite and does
not include durable I/O or macOS sealing cost.

The recall-forensics evidence is `recall-forensics.json`. It is generated by:

```bash
python3 paper/scripts/run_recall_forensics_eval.py \
  --filler 1000 \
  --seed 20260716 \
  --output paper/data/recall-forensics.json
```

Its reconstruction, eligible-turn re-execution, and mutation-detection counts
are deterministic for the source manifest, fixed workload, and seed; only
`reconstruction_ms_total` is a single-host timing observation. Three turns
that precede correction or deletion are reconstructed from evidence but are
not counted as engine re-executions. The forensic checks use only Python
standard-library modules, including `sqlite3` and `hashlib`; the workload and
eligible-turn re-execution phases import aetnamem. `paper/scripts/
validate_evidence.py` checks the source-file digests, JSON values, and
manuscript table.

```bash

```
