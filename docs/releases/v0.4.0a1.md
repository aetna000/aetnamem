# AetnaMem 0.4.0a1 — Evidence to approved change

Released: 2026-07-20  
Status: alpha

AetnaMem 0.4.0a1 adds a provider-neutral collaborative decision kernel and an
Evidence-to-Decision (EtD) profile. It connects evidence, panel judgments,
recommendations, institutional approvals, scoped authorization, and guarded
implementation without turning AetnaMem into a hospital user-management
system or coupling it to one model provider.

## Why this release

Organizations rarely fail because they have no evidence. They fail at the
handoffs: which evidence informed a judgment, who was eligible to vote, how a
recommendation was adopted, who approved resources, and whether the action
that ran was exactly the action authorized.

This release makes those transitions explicit, versioned, auditable, and
host-embeddable. A hospital, policy group, or business can install the library
inside its own authenticated server while keeping identity, UI, compliance,
and deployment decisions under host control.

## What is included

- Generic decision cases with pinned templates, immutable artifact revisions,
  exact evidence links, roles, capabilities, conflicts, and recusal.
- Frozen ballot eligibility, hidden-until-close voting, immutable vote
  revisions, deterministic quorum/consensus outcomes, idempotency, and
  optimistic concurrency.
- Separate recommendation adoption, institutional approval, scoped change
  authorization, and guarded action execution.
- SQLite for single-server use and PostgreSQL for horizontal/multi-process
  hosts, including real process-level vote and close-race contract tests.
- Optional Ed25519 or host-supplied AWS KMS principal attestations and signed
  outcome, adoption, approval, authorization, and purge receipts.
- Independent payload and conflict-of-interest retention policies with
  digest-bound logical purge receipts.
- Clinical and generic EtD templates, deterministic Markdown reporting, an
  offline bundle verifier, a complete playground, and an external pilot/
  methodology-review runbook.
- No change to default `Memory` or memory-only MCP behavior. Claude, Grok/xAI,
  OpenAI, Ollama, and non-LLM hosts can use the same Python contracts.

## Install

```bash
pip install --pre aetnamem==0.4.0a1

# PostgreSQL only
pip install --pre 'aetnamem[postgres]==0.4.0a1'

# PostgreSQL + Ed25519 signing
pip install --pre 'aetnamem[production]==0.4.0a1'
```

Run the complete local workflow:

```bash
aetnamem-etd-playground \
  --db ./etd-playground.db \
  --output ./etd-playground-output

aetnamem-etd-verify ./etd-playground-output/decision-bundle.json
```

Run the same workflow on PostgreSQL without putting the DSN in command-line
history:

```bash
export DECISION_DATABASE_URL='postgresql://...'
aetnamem-etd-playground \
  --postgres-dsn-env DECISION_DATABASE_URL \
  --output ./etd-playground-output
```

## Minimal host example

```python
from aetnamem.decisions import ActorContext, DecisionEngine
from aetnamem.etd import clinical_etd_template

engine = DecisionEngine.postgres(database_dsn)
chair = ActorContext(
    namespace_id=authenticated_organization_id,
    principal_id=authenticated_user_id,
)

case = engine.create_case(
    chair,
    title="Medication reconciliation",
    template=clinical_etd_template(),
    content={"question": "Should we introduce pharmacist reconciliation?"},
    idempotency_key=request_id,
)
```

The host must derive actor fields from authenticated server state. It should
never accept tenant or principal identity directly from an untrusted payload.

## Verification

The release was validated with 173 tests. These include eight simultaneous
PostgreSQL voter processes, a vote-versus-close process race, the complete EtD
chain through scoped authorization, signed receipt verification, retention
purge, clean wheel/sdist installation, and the existing memory/action suite.

## Important boundaries

This is an alpha SDK, not a hosted user layer. It does not claim GRADE
compliance, clinical validity, regulatory compliance, cryptographic secret
ballots, or that an external EtD pilot has already occurred. The repository
contains a complete pilot and independent-review protocol; a real institution
and an independent methodologist must generate that external evidence.

See the [decision workflow specification](../decision-workflow-spec.md),
[host integration guide](../decision-host-integration.md), and [pilot and
methodology-review runbook](../etd-pilot-methodology-review.md).
