# AetnaMem 0.4.1 — Cache-aware memory for agent hosts

Released: 2026-07-21  
Status: stable Python and OpenClaw integration release

AetnaMem 0.4.1 makes the cache-aware memory layout a provider-neutral public
contract and ships the validated OpenClaw adapter as npm 0.2.4. Agent hosts can
now request bounded stable persona context and selective turn-specific recall
without depending on an OpenClaw, Hermes, or model-provider SDK.

## What is included

- Python `Memory.build_context_pack(...)`.
- CLI `aetnamem context-pack ...`.
- MCP `memory_context_pack`, returning `aetnamem-context-pack-v1` with stable
  and dynamic blocks, placement hints, hard budgets, SHA-256 bindings, and full
  audit record IDs; dynamic recall does not duplicate records already present
  in the stable block.
- OpenClaw npm plugin 0.2.4 with cache-aware prompt placement, compact
  model-visible references, conservative defaults, and optimized tool schema.
- A dedicated Hermes MCP and automatic-adapter guide plus a generic Python
  adapter example.
- Reproducible three-arm OpenClaw 2026.7.1-2 / DeepSeek V4 Flash benchmark,
  raw provider usage, audit verification, and documented limitations.

## Install

```bash
pip install --upgrade aetnamem
```

For OpenClaw:

```bash
openclaw plugins install npm:openclaw-memory-aetnamem@latest --pin
openclaw aetnamem setup --single-user --subject you
```

For any MCP host:

```bash
aetnamem mcp --db ~/.aetnamem/memories.db --subject you
```

## Measured result and its boundary

Across 20 fresh-session tasks per arm, native `MEMORY.md` used 596,581 prompt
tokens, the previous AetnaMem layout used 521,858, and cache-aware AetnaMem
used 517,118. The cache-aware arm used 13.320% fewer prompt tokens and cost
2.968% less than native with 20/20 correct answers in every arm. Compared with
the previous AetnaMem layout it saved 0.908% of prompt tokens and 1.190% of
cost.

Absolute cache reads did not improve over the previous AetnaMem layout. The
incremental benefit came from a smaller model-visible surface, so this release
does not claim that placement alone creates cache hits or that another host or
provider will reproduce the percentage. See the
[evaluation report](../openclaw-memory-evaluation.md) and
[raw benchmark artifacts](../../bench/openclaw_memory/results/) before quoting
the result.

## Compatibility and security

The memory core remains provider-neutral and preserves existing recall,
capture, deletion-receipt, audit, SQLite, PostgreSQL, and EtD behavior. The
context pack is prompt material, not an authorization boundary. Multi-user
hosts must authenticate callers, derive subject scope outside the model, and
enforce database authorization server-side.
