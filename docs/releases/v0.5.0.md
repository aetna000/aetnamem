# AetnaMem 0.5.0 — Four memories, one runtime

Status: **public release; CML modes remain experimental and default off**

> **AetnaMem remembers whether remembering actually helped.**

AetnaMem 0.5.0 adds an opt-in orchestration layer for working, semantic,
episodic, and procedural memory while preserving the existing Python
`Memory`, CLI, and default 15-tool MCP contracts.

## Why this release

Agents already have fragments of all four memory types, but they are normally
managed separately: current task state in the context window, facts in
`MEMORY.md`, past attempts in conversation or trace history, and procedures in
`SKILL.md`. That makes setup difficult, repeats context, and gives operators no
consistent record of which memory was used.

v0.5.0 coordinates those sources behind one bounded runtime connection:

| Memory | Typical agent location | AetnaMem representation | Example |
|---|---|---|---|
| Working | Context window or task scratchpad | `working_snapshots` in `memory.db` | “Motor 12 inspected; oil check pending.” |
| Semantic | `MEMORY.md`, profile, or fact store | Governed records and episodes in `memory.db` | “Motor 12 requires ISO VG 68 oil.” |
| Episodic | Conversation history and execution traces | `experience_outcomes` and reviewed lessons in `memory.db` | “The last repair failed because the wrong oil was used.” |
| Procedural | `SKILL.md`, runbook, or tool instructions | Versioned procedure index in `memory.db`; source file remains on disk | “Isolate, inspect, lubricate, test, and document.” |

The second reason is measurement. Retrieval does not prove usefulness. The
experimental **Causal Memory Ledger**—“causal” means cause and effect—records
which eligible memory was assigned, which was actually shown, and which
outcome followed. Its user-facing idea is **Memory Impact**:

> Which memories actually helped the agent, and which only consumed context?

This release provides the measurement foundation. It does not yet claim a
proven causal improvement.

## Install

```bash
python3 -m pip install --upgrade aetnamem
openclaw plugins install npm:openclaw-memory-aetnamem@latest --pin
aetnamem setup
openclaw aetnamem setup --single-user --subject you \
  --orchestrated --runtime-config ~/.aetnamem/runtime.json
```

## Added

- `MemoryRuntime` under `aetnamem.runtime`.
- Additive runtime, working, experience, lesson, procedure, and manifest
  tables in the existing SQLite file.
- `starter`, `private`, `team`, and `benchmark` configurations.
- A ten-step `aetnamem setup` wizard.
- `aetnamem runtime` commands for initialization, validation, preparation,
  outcomes, lesson promotion, deletion, health, and MCP.
- `memory_prepare_turn` and `memory_record_outcome` on the opt-in runtime MCP
  endpoint.
- Caller-asserted outcome capture through generic CLI/MCP, quarantined lesson
  proposals, versioned `SKILL.md` selection, idempotent outcomes, and
  four-plane deletion receipts.
- A deterministic leave-one-plane-out benchmark.
- OpenClaw plugin 0.3.0 with capability detection, opt-in orchestration,
  legacy fallback, and configuration-preserving setup writes.
- A default-off experimental CML foundation with shadow and benchmark-only
  experiment modes, pre-compilation intervention records, runtime-pack v2,
  exact manifest binding, structured metrics, and explicit outcome trust
  labels.

## Compatibility

- `aetnamem mcp` keeps the original catalog and behavior.
- `aetnamem runtime mcp` adds the two orchestration tools after that catalog.
- Existing OpenClaw configurations remain on legacy recall unless
  `orchestration.enabled` is explicitly set.
- New OpenClaw code falls back to legacy recall when connected to Python
  versions without runtime tools.
- Runtime schema changes are additive; existing semantic records, audit
  hashes, and public v1 receipts are not rewritten.

## Deployment boundary

The reference runtime is embedded and local-first. It provides scoped
provider contracts and one public MCP endpoint. A production distributed
deployment still needs host-supplied authentication, networking,
authorization, and a suitable concurrent backend.

## CML claims boundary

This release implements the intervention ledger, not the research result.
It does not yet ship causal estimators, confidence intervals, the planted-effect
benchmark, a completed Grok CLI study, or a held-out outcome-per-cost policy.
No release claim should say that CML has improved agent quality or cost until
those falsifiable gates pass. See
[current capability status](../current-status.md) and [`plan.md`](../../plan.md).
